Motivation

Predictive maintenance is one of the highest-value applications of machine learning in DACH industry. The market is anchored by Siemens, MTU Aero Engines, Lufthansa Technik, ZF, and Bosch, all of whom run continuous instrumentation programmes on rotating machinery. Replacing fixed-interval maintenance with condition-based RUL prediction reduces unscheduled downtime and lowers spare-part inventories, with documented total-cost reductions of 20 to 40 percent in heavy industry.

Why C-MAPSS?

NASA's Commercial Modular Aero-Propulsion System Simulation is the de-facto academic benchmark for data-driven prognostics. It simulates run-to-failure trajectories of a 90,000 lb-thrust turbofan and underpins more than a decade of comparable studies. FD001 is the simplest of four subsets and the canonical entry point.

Project 11 is the analytical twin of project 7 (Industrial Anomaly Detection). The two share a domain but differ in framing: anomaly detection produces a binary signal at each step; RUL produces a continuous regression. In a real deployment the two compose: anomaly detection triggers a closer look, and RUL regression supplies the lead time for maintenance scheduling.

Schema

Each row: unit_number, time_in_cycles, three operational settings, 21 noisy sensor measurements. Each RUL_FDxxx.txt is a single column of remaining cycles for the test units.

Phase 1 status

Code-only scaffold. data/README.md documents the Kaggle CLI command. Actual download and unzip happens in Phase 2.

Methods

Preprocessing

  1. Drop sensors with training-set standard deviation below 1e-3 (constant or near-constant).
  2. Z-score remaining signals using training statistics; apply unchanged to test.
  3. Build piece-wise linear RUL labels capped at 125 cycles (Heimes 2008, Zheng 2017).

Train and validation split

Hold out 20 percent of the 100 training engines via fixed-seed GroupShuffleSplit. Critically, the split is by engine, not by row, to prevent within-engine leakage.

Metrics

Three metrics are reported on both validation and the NASA test set:

  • RMSE in cycles - direct error magnitude.
  • MAE in cycles - mean operational cost proxy.
  • NASA score - asymmetric exponential, penalises late predictions more heavily. sum(exp(-d/13)-1) for d<0, sum(exp(d/10)-1) for d>=0.

Baseline: Random Forest on lag and window features

Per-cycle features over the trailing 30 cycles of each retained signal: mean, standard deviation, minimum, maximum, OLS slope. Approximately 85 features per cycle on FD001.

Why a tree baseline still matters

  • Random Forest with 30-cycle window stats often reaches within 10-15% of LSTM RMSE on FD001.
  • Feature importance is human-readable: a reliability engineer can sanity-check whether the model has latched onto sensors that domain physics suggests should matter.
  • When baseline and LSTM disagree, that is a diagnostic - large gaps suggest insufficient feature engineering rather than insufficient model capacity.

Advanced: LSTM and 1D-Transformer on raw sequences

Sliding window of length 30 over standardised raw signals. Approximately 20,000 training windows on FD001. Two architectures share the same script:

LSTM (default)

  • 2 LSTM layers, hidden size 64, dropout 0.2.
  • ReLU head, scalar output.
  • Adam, learning rate 1e-3, batch size 256, MSE loss.
  • Early stop on validation RMSE, patience 10.

1D-Transformer (alternative)

  • Sinusoidal positional encoding.
  • Model dim 64, 4 attention heads, 2 encoder layers.
  • Same head and optimiser as the LSTM.
  • Switch via MODEL=transformer python src/model_advanced.py.
Phase 1 status

Both architectures live in src/model_advanced.py. Convergence curves and final test metrics are placeholders pending Phase 2 execution. Whether the transformer beats the LSTM at our parameter budget is intentionally not pre-decided.

Discussion

Strengths of the scaffold

  • Clean separation between Phase 1 (scaffolding) and Phase 2 (execution). Numbers cannot leak in by accident.
  • Identical script structure across baseline and advanced for direct comparison.
  • Three metrics (RMSE, MAE, NASA score) prevent gaming of any single metric.

Limitations

  • FD001 is the easiest subset. FD002 and FD004 require per-regime normalisation (six operating conditions).
  • RUL cap of 125 cycles is a heuristic; sensitivity analysis is appropriate.
  • No uncertainty quantification yet. Bayesian deep learning (Li 2021) is a natural extension.
  • Test evaluation uses only the final cycle of each test engine, matching NASA protocol but discarding earlier cycles.
  • C-MAPSS is a simulation; transfer to real fleet data needs domain adaptation (Listou Ellefsen 2019, Singh 2025).

Industrial deployment considerations

  • Edge versus cloud: model size under 1 MB is compatible with both.
  • Drift monitoring: deployed models need a feedback loop to detect input drift and trigger re-training.
  • Explainability: regulators and crews want explanations - tree feature importance is built-in; SHAP for the deep model.
  • Certification: EASA and FAA are still developing ML-prognostics guidance. Production needs a model card, training-data lineage, and re-training schedule.

Conclusion

We deliver a complete Phase 1 scaffold for NASA C-MAPSS RUL prediction following the Liora cohort 6973 layout. The deliverable comprises:

  • A Random Forest baseline on 30-cycle window statistics.
  • An LSTM (with optional 1D-Transformer) on raw sensor sequences.
  • 33 verified references, validated against CrossRef and Europe PMC on 2026-05-08.
  • An exploratory data analysis notebook (raw, not executed).
  • A 4000+ word IMRaD manuscript with placeholder numbers explicitly flagged.
  • This self-contained HTML presentation.

Phase 2 will populate placeholders by re-running the same scripts against actual C-MAPSS data and injecting canonical numbers from the resulting JSON artefacts. Future extensions span domain adaptation across the four C-MAPSS subsets, Bayesian uncertainty quantification, and survival-analysis comparison via Random Survival Forests (Ishwaran 2008).
